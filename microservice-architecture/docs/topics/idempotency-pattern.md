# Idempotency Pattern

## 场景问题

在分布式系统里，重复请求并不是边缘情况，而是常态来源之一：

- 前端用户重复点击提交按钮
- 网关或客户端因为超时发起重试
- 消息队列发生重复投递
- 下游调用方不知道上次到底是“成功了还是超时了”

如果系统没有幂等控制，就会出现重复扣款、重复下单、重复库存预占等严重副作用。

## 核心思想

幂等模式的目标不是“拦截所有重复流量”，而是：

- 对同一个业务请求分配稳定的幂等键
- 第一次请求真正执行业务逻辑并保存结果
- 后续相同幂等键的重复请求直接返回首次结果
- 如果同一幂等键对应的业务参数发生变化，则返回冲突而不是误认为同一请求

## 结构与职责

```text
Client
  -> order-service
     - check idempotency key
     - compare request fingerprint
     - create order once
     - cache result
```

常见数据项：

- `idempotencyKey`
- `requestFingerprint`
- `responseSnapshot`
- `status`

## 最小可运行示例

本专题的最小示例统一表达 3 条路径：

1. 首次请求：创建订单，状态 `CREATED`
2. 重复请求且业务参数一致：返回第一次结果，并标记 `replayed = true`
3. 重复请求但业务参数不同：返回 `CONFLICT`

统一业务语境：

- 幂等键：`IDEMP-ORDER-1001`
- 订单：`ORD-1001`
- 商品：`SKU-BOOK`

## 进阶演进

真实系统里，幂等模式通常进一步演进为：

1. 将幂等记录落库或落 Redis
2. 结合过期时间控制存储成本
3. 结合事务保证“业务写入”和“幂等写入”原子化
4. 消费端幂等与 API 幂等分开治理
5. 结合 Outbox / 消息去重实现端到端一致性

## 常见误区

1. 只记录幂等键，不比较请求参数
2. 只做接口幂等，不做消息消费幂等
3. 把幂等和锁混为一谈
4. 幂等记录没有过期策略，导致无限增长
5. 失败结果不落记录，导致重试期间行为不稳定

## 推荐和哪些模块一起看

- `service-communication`
- `resilience-patterns`
- `configuration-center`

原因：

- 通信方式决定重复请求从哪里来
- 重试会放大重复请求概率
- 配置中心可用于统一幂等 TTL、冲突策略和缓存开关
