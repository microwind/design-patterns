# API Gateway

## 场景问题

当系统拆成多个服务之后，客户端如果直接访问各个服务，会遇到几个问题：

- 入口分散，客户端需要记住多个地址
- 鉴权、限流、请求头透传等逻辑会在每个服务重复实现
- 前端往往不关心内部服务拓扑，只想访问一个稳定入口

## 核心思想

API Gateway 的目标不是替代业务服务，而是提供统一入口和通用横切能力：

- 路由转发
- 鉴权校验
- 统一响应
- 请求头透传
- 聚合能力

## 结构与职责

```text
Client
  -> API Gateway
     - 路由匹配
     - 鉴权
     - 透传上下文
     -> order-service
     -> inventory-service
```

## 最小可运行示例

Go 示例实现了一个最小网关：

- 根据路径前缀把请求转发到不同服务处理器
- 校验 `X-User` 请求头
- 自动补齐 `X-Correlation-ID`
- 对未知路由返回 `404`

参考目录：

- `examples/api-gateway/go`

## 进阶演进

下一步通常会继续增加：

1. 统一限流
2. JWT/OAuth2 鉴权
3. 灰度路由
4. API 聚合
5. 统一可观测性埋点

## 常见误区

1. 把所有业务编排都堆进网关
2. 网关直接操作业务数据库
3. 把网关当 BFF、ESB、工作流引擎的混合体
4. 没有给网关本身设计高可用和限流
