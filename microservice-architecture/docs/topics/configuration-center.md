# Configuration Center

## 场景问题

微服务进入多环境、多实例部署后，配置会迅速变成一个系统级问题：

- 不同环境有不同数据库地址、超时阈值、功能开关
- 多个服务实例如果各自维护本地配置文件，容易出现配置漂移
- 配置改动如果必须重新发版，交付效率会很差
- 没有版本和来源管理时，很难追查线上配置到底是谁改的

## 核心思想

配置中心的目标不是“多一个配置文件”，而是把运行时配置从应用代码和部署镜像里解耦出来，变成一类可集中管理、按环境隔离、可刷新、可追踪的基础设施能力。

典型职责包括：

- 按 `service + environment` 存储配置
- 支持客户端读取当前配置快照
- 支持配置版本号
- 支持配置变更后的刷新
- 为后续灰度、审计和回滚打基础

## 结构与职责

```text
Config Center
  - 存储 order-service@prod / inventory-service@prod 配置
  - 维护配置版本
  - 返回最新配置快照

Config Client
  - 启动时加载配置
  - 保存本地快照
  - 按需刷新

Order Service
  - 使用 timeoutMs / featureFlags / dbHost 等配置
```

## 最小可运行示例

本专题的最小示例统一表达 3 件事：

1. 配置中心按 `serviceName + environment` 存储配置
2. 客户端首次加载得到配置快照
3. 配置更新后客户端再次刷新，读取到新版本配置

为了让模式本身更清晰，示例不引入真实网络协议，而是先用进程内模拟配置中心。

## 进阶演进

真实系统里，配置中心通常进一步演进为：

1. 配置分层：公共配置、服务配置、环境配置
2. 热更新：推送刷新或长轮询
3. 权限与审计：谁能改、谁改了什么、何时改的
4. 灰度配置：按集群、标签或租户差异化下发
5. 配置校验：值域、格式和兼容性检查

## 常见误区

1. 把所有配置都塞进环境变量，导致治理能力不足
2. 业务代码里散落硬编码常量和本地配置文件
3. 修改配置后没有版本号和来源记录
4. 配置中心只负责“存”，不负责校验和回滚策略

## 推荐和哪些模块一起看

- `service-discovery`
- `load-balancing`
- `resilience-patterns`

原因：

- 服务发现解决“去哪里找服务”
- 配置中心解决“服务运行时参数从哪里来”
- 韧性模式解决“拿到配置之后如何用它控制超时、重试和熔断”
