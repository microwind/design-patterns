# Service Communication

## 场景问题

服务拆分之后，最大的现实问题之一就是：服务之间到底怎么通信。

常见选择有：

- REST / HTTP
- RPC
- MQ / Event Bus

不同通信方式的差别，不只是协议不同，而是：

- 耦合程度不同
- 时效性不同
- 一致性模型不同
- 故障传播方式不同

## 核心思想

可以把服务通信简单分成两类：

- 同步通信：请求方等待响应再继续
- 异步通信：请求方发出事件，后续处理延后完成

同步强调实时性，异步强调解耦和削峰。

## 结构与职责

```text
同步：
Order Service -> Inventory Service -> Payment Service

异步：
Order Service -> Event Bus -> Inventory Handler / Payment Handler
```

## 最小可运行示例

Go 示例同时包含两套流程：

- 同步版：订单服务串行调用库存与支付
- 异步版：订单服务发布事件，由事件总线驱动库存与支付处理

这样可以直接对比：

- 成功路径
- 失败路径
- 状态推进方式

参考目录：

- `examples/service-communication/go`

## 进阶演进

1. 引入消息队列替代进程内事件总线
2. 设计事件 schema 与版本演进
3. 增加幂等键与死信队列
4. 将异步流程升级为 Saga

## 常见误区

1. 以为异步天然更高级
2. 同步链路过长，导致尾延迟放大
3. 异步系统没有状态跟踪与补偿机制
4. 混用同步和异步，却没有清晰边界
