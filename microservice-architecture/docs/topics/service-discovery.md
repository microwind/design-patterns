# Service Discovery

## 场景问题

微服务实例通常是动态变化的：

- 服务可能扩容
- 服务可能缩容
- 实例可能故障下线
- 部署后地址可能变化

如果调用方把地址写死在配置里，会导致系统难以扩缩容，也难以自动恢复。

## 核心思想

服务发现的目标是：

- 让服务实例先向注册中心登记
- 让调用方通过逻辑服务名发现实例
- 让地址变化从业务代码里抽离出去

## 结构与职责

```text
inventory-service instance-A ----\
inventory-service instance-B ----- > Service Registry
order-service client -----------/
```

调用流程：

1. 服务实例注册自己
2. 调用方按服务名查询实例列表
3. 再结合负载均衡策略选出目标实例

## 最小可运行示例

Go 示例实现了一个内存版注册中心：

- `Register`
- `Deregister`
- `Instances`
- `Next`（轮询选择实例）

它并不追求生产级能力，而是把服务发现机制讲清楚。

参考目录：

- `examples/service-discovery/go`

## 进阶演进

1. 增加心跳与过期摘除
2. 增加健康检查
3. 增加元数据与标签路由
4. 和配置中心、负载均衡联动

## 常见误区

1. 只做注册，不做摘除
2. 把注册中心当数据库长期存业务数据
3. 发现到实例列表后，不做本地缓存与刷新策略
4. 忽略实例元数据，导致灰度和多机房路由困难
