# API Versioning

## 场景问题

微服务接口一旦被多个客户端依赖，就不能再随意“直接改掉旧字段”。否则：

- 老客户端会被新版本破坏
- 联调和发布窗口被强耦合
- 服务升级会演变成全链路协调成本

## 核心思想

API 版本控制的核心不是“多写一个 v2 路径”，而是：

- 兼容旧调用方
- 支持新能力逐步上线
- 让演进可控，而不是一次性替换

## 结构与职责

```text
Client
  -> Version Router
     - 默认走 v1
     - 根据 Header/Path 选择版本
     -> ProductHandlerV1
     -> ProductHandlerV2
```

## 最小可运行示例

Go 示例中包含：

- 默认版本 `v1`
- 通过 `X-API-Version` 选择 `v2`
- 不同版本返回不同字段结构
- 对不支持的版本返回错误

参考目录：

- `examples/api-versioning/go`

## 进阶演进

真实系统里，版本控制通常进一步包含：

1. Header 版本策略与路径版本策略比较
2. 版本废弃公告
3. 灰度流量迁移
4. 契约测试
5. OpenAPI 文档版本管理

## 常见误区

1. 新旧版本直接复用一套返回结构，结果谁都改不动
2. 每次小改动都新开一个版本，导致版本爆炸
3. 不提供默认版本，客户端必须强依赖版本细节
4. 没有下线策略，旧版本永久堆积
