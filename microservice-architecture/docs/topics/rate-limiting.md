# Rate Limiting

## 场景问题

高峰流量或恶意流量出现时，如果系统对所有请求都无条件放行，下游很容易被拖垮。

典型后果：

- 线程池被打满
- 数据库连接耗尽
- 正常用户请求被噪声流量挤掉

## 核心思想

限流的目标不是“尽量多挡”，而是在容量边界内有选择地放行请求。

最小示例采用固定窗口限流：

- 每个窗口允许最多 N 次请求
- 超过阈值的请求返回拒绝
- 窗口推进后重新开始计数

## 最小可运行示例

统一模型：

1. 前 3 次请求允许通过
2. 第 4 次请求被拒绝
3. 窗口推进后再次允许

## 常见误区

1. 把限流和熔断混为一谈
2. 只在网关做限流，忽略服务内热点保护
3. 限流没有配套降级策略
