# Distributed Transactions

## 场景问题

在微服务里，一个业务动作往往跨多个服务：

- 订单服务创建订单
- 库存服务预占库存
- 支付服务发起扣款

如果执行到一半失败，就会出现中间态：

- 订单已创建，但库存没锁住
- 库存锁住了，但支付失败
- 支付成功了，但订单状态没更新

这类问题不能靠单库事务直接解决。

## 核心思想

分布式事务的目标不是让所有服务像单机事务那样“一把锁住”，而是让跨服务步骤能够最终收敛到业务可接受状态。

本专题的最小示例使用 Saga 补偿事务来表达：

1. 先执行本地事务
2. 某一步失败时执行补偿动作
3. 最终把状态推进到 `COMPLETED` 或 `CANCELLED`

## 结构与职责

```text
Order Saga
  1. create order (PENDING)
  2. reserve inventory
  3. charge payment
  4. success -> COMPLETED
  5. failure -> compensate -> CANCELLED
```

补偿动作示例：

- 库存已预占但支付失败 -> 释放库存
- 订单已创建但库存失败 -> 取消订单

## 最小可运行示例

本专题的最小示例统一表达两条路径：

1. 成功路径：订单创建 -> 库存预占 -> 支付成功 -> `COMPLETED`
2. 失败路径：订单创建 -> 库存预占 -> 支付失败 -> 释放库存 -> `CANCELLED`

统一业务语境：

- 订单：`ORD-1001`
- 商品：`SKU-BOOK`
- 状态：`PENDING / COMPLETED / CANCELLED`

## 进阶演进

真实系统里，分布式事务通常会进一步演进为：

1. Saga 编排式与协同式对比
2. TCC
3. 2PC / XA 在特定场景下的权衡
4. 结合 Outbox 发布 saga 事件
5. 结合幂等与重试做补偿动作稳定化

## 常见误区

1. 以为引入消息队列就自动解决一致性
2. 只设计正向流程，不设计补偿动作
3. 把技术事务和业务补偿混为一谈
4. 补偿动作本身不做幂等
5. 没有清晰状态机，导致恢复路径不可推理

## 推荐和哪些模块一起看

- `service-communication`
- `outbox-pattern`
- `idempotency-pattern`

原因：

- 通信方式决定 saga 如何驱动
- outbox 让事务事件可靠发出
- 幂等保证补偿与重试不会制造新的副作用
