# Load Balancing

## 场景问题

当同一个服务存在多个实例时，请求不能只打到一个节点，否则：

- 单实例容易成为瓶颈
- 某个实例故障会放大影响面
- 资源利用率不均衡

## 核心思想

负载均衡的核心是：在一组可用实例中，按照既定策略选择下一个请求目标。

常见策略包括：

- 轮询
- 加权轮询
- 最少连接
- 一致性哈希

## 结构与职责

```text
Client / Gateway
  -> Load Balancer
     -> instance-A
     -> instance-B
     -> instance-C
```

负载均衡器本身不处理业务，它只负责“选谁”。

## 最小可运行示例

Go 示例提供了 3 种策略：

- `RoundRobinBalancer`
- `WeightedRoundRobinBalancer`
- `LeastConnectionsBalancer`

测试中会验证：

- 轮询是否交替选择实例
- 加权策略是否体现权重差异
- 最少连接是否优先选择空闲节点

参考目录：

- `examples/load-balancing/go`

## 进阶演进

1. 结合健康检查摘除故障节点
2. 结合服务发现动态刷新实例列表
3. 对热点用户使用一致性哈希
4. 结合网关或 Sidecar 完成流量治理

## 常见误区

1. 以为“有多个实例”就等于“系统高可用”
2. 不做健康检查，让故障实例继续接流量
3. 忽略实例能力差异，却使用平均分配策略
4. 用最少连接策略时不维护连接状态
